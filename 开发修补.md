# 聚发舟 MultiPost AI v3 版本优化开发与 PR 提交流程

## 一、v3 版本目标

v3 版本不做真实平台登录，不做真实发帖，严格围绕比赛题目实现：

1. 用户输入内容；
2. AI 自动改写；
3. 自动适配公众号、知乎、B站、小红书；
4. 支持图片、视频、封面上传；
5. 支持多平台预览编辑；
6. 支持一键模拟发布；
7. 支持发布记录、详情查看和删除；
8. 支持扩展更多平台的架构设计；
9. 使用后端数据库持久化数据；
10. 保证每个功能通过单独 PR 持续提交。

---

# 二、整体开发方式

每个模块都按这个流程做：

```text
从 main 拉取最新代码
↓
新建 feature 分支
↓
让 AI 修改当前模块代码
↓
本地运行测试
↓
提交 commit
↓
推送分支
↓
创建 PR
↓
合并到 main
↓
开始下一个模块
```

注意：

```text
不是最后统一合并。
而是每完成一个模块，就合并一次 PR 到 main。
```

这样可以形成持续开发记录。

# 三、每个模块如何让 AI 修改

每次找 AI 修改代码时，建议按下面格式提问：

```text
我正在开发仓库 https://github.com/1781988/multiplatform.git

当前要修改的模块是：【模块名称】

当前问题：
1. xxx
2. xxx

目标效果：
1. xxx
2. xxx

请你基于当前项目结构，告诉我需要修改哪些文件，并直接给出完整代码或关键代码。
要求：
1. 不影响已有功能；
2. 保持前后端接口一致；
3. 修改后可以本地运行；
4. commit 备注使用中文；
5. 当前模块完成后我会单独提交一个 PR。
```

如果 AI 给的代码太散，就继续问：

```text
请你不要只给片段，请按文件路径分别给出完整修改内容。
```

如果出现报错，就问：

```text
这是我运行后的报错，请你根据报错定位问题，并告诉我具体修改哪个文件。
```

# 五、每个 PR 如何合并

## 方式一：GitHub 页面合并

1. 打开仓库：

```text
https://github.com/1781988/multiplatform
```

2. 进入 Pull requests；
3. 点击 New pull request；
4. base 选择：

```text
main
```

5. compare 选择当前分支；
6. 填写 PR 标题和描述；
7. 创建 PR；
8. 确认无冲突后点击：

```text
Merge pull request
```

9. 点击：

```text
Confirm merge
```

10. 回到本地执行：

```bash
git checkout main
git pull origin main
```

然后继续下一个模块。

## 方式二：命令行合并

不推荐新手用，但也可以：

```bash
git checkout main
git pull origin main
git merge feature/v3-records-layout
git push origin main
```

比赛更推荐用 GitHub PR 页面合并，因为会留下清晰 PR 记录。

# 七、最终 main 分支状态

最后所有 PR 合并后：

```text
main = 完整 v3 项目
```

所有临时分支只是为了形成开发记录和功能隔离。最终评委主要看：

```text
main 分支是否可运行
PR 是否持续
commit 是否清晰
README 是否完整
```

因此你每做完一个模块，都要合并回 main，不要等最后统一合并。
